# Climbing Coach

An iPhone app (Expo / React Native) that analyzes your **climbing technique**
from a phone video — not by comparing you to a pro, but by scoring
climbing-specific biomechanics: balance, hip positioning, flagging, reach
efficiency, stability, dynamic control, smoothness, and route adherence.

> This is a **working MVP**, not a mockup. Every arrow in the diagram below
> is implemented. Where a component is honestly an approximation (e.g. the
> pseudo-3D lift, the stylized body mesh) it is clearly labeled in code
> AND in this README.

```
 ┌───────────┐   ┌───────────┐   ┌──────────────┐   ┌──────────┐   ┌──────────────┐
 │  Video    │ → │  2D pose  │ → │ Pseudo-3D    │ → │ Phase    │ → │  Technique   │
 │ (record/  │   │ inference │   │   lift       │   │ segment  │   │  scoring +   │
 │  upload)  │   │ (provider)│   │ (heuristic)  │   │ (rules)  │   │  coaching    │
 └───────────┘   └───────────┘   └──────────────┘   └──────────┘   └──────────────┘
                       ▲                                                  │
                       └──────── tagged holds ────────────────────────────┘
```

## What the app does

1. **Record** a climb in-app, or **upload** a prerecorded clip.
2. **Tag the holds** on the reference frame — tap to add, pick type and role,
   arrange the intended sequence.
3. The app runs the analysis pipeline (pose → 3D lift → phases → scoring).
4. You get a synchronized analysis screen:
   - video + real 2D keypoint overlay (synced to the scrubber)
   - 3D stick skeleton + stylized body mesh (orbiting camera)
   - phase timeline
   - per-category score grid
   - coaching tips with explanations
5. Sessions are stored locally; a sparkline on the Home screen shows
   technique progression over time.

## What is real vs approximated

| Component               | Status     | Notes                                                                 |
| ----------------------- | ---------- | --------------------------------------------------------------------- |
| Video recording         | Real       | `expo-camera` CameraView, saved to app FS.                            |
| Video upload            | Real       | `expo-image-picker` + thumbnail extraction.                           |
| Manual hold tagging     | Real       | Normalized coordinates, hold type/role/sequence editor, undoable.     |
| 2D pose inference       | Pluggable  | `MockPoseProvider` (seeded, demo-ready) OR `TFJSPoseProvider` (real, needs dev build). |
| Pseudo-3D lift          | Heuristic  | Depth solved from anthropometric bone lengths + climbing priors. *Not* a learned monocular 3D pose model — deliberately interpretable. |
| Phase segmentation      | Real       | Rule-based state machine over contact count / CoM motion / flag geometry. |
| Scoring engine          | Real       | 8 interpretable categories, each a pure function with a rationale.    |
| 3D stick skeleton       | Real       | `expo-gl` + `three`, updates per-frame from lifted joints.            |
| Stylized body mesh      | Approx.    | Capsule-per-limb + torso box. Flagged in source; upgrade path is local. |
| Route-aware coaching    | Real       | Scores use tagged holds, support polygon, and intended sequence.      |
| Session history / progress | Real    | SQLite (device) / in-memory (tests + Expo Go fallback).               |

Sessions produced by the seeded mock pose provider are badged **Demo** in
the UI.

## Setup

Requirements: Node ≥ 20, Xcode 16+ (for iOS builds), iPhone or iOS simulator.

```bash
npm install
npm run typecheck
npm test
npm start                 # Expo Go compatible (uses MockPoseProvider)
```

For real pose inference (MoveNet via tfjs-react-native):

```bash
npx expo prebuild --platform ios
npm install @tensorflow/tfjs @tensorflow/tfjs-react-native \
            @tensorflow-models/pose-detection react-native-fs
npx expo run:ios
```

The `TFJSPoseProvider` is stubbed — it declares the interface and fails with
a clear `NEEDS_DEV_BUILD` error when native deps are missing, letting the
runtime fall back to the mock provider automatically. Finishing the TFJS
integration is a localized change (see `src/analysis/pose/TFJSPoseProvider.ts`).

## Demo

On first launch the app **seeds itself** with:

- a demo user
- a 6-hold "Demo V3: Left Traverse" route
- one fully-analyzed session, generated by running the *real* pipeline on
  the mock pose trace

Open the Sessions tab → tap the demo session → explore the synced video,
3D skeleton, phase timeline, scores, and coaching tips.

## Running the tests

```bash
npm test                  # 22 tests across domain / analysis / storage
npm run test:ci           # with coverage
```

Tests run in Node with zero native dependencies. The storage layer has a
tiny in-memory SQL fake that handles the exact subset of SQL this app
emits — keeps CI fast and deterministic.

## Known limitations (V1)

- Pseudo-3D depths are scale-ambiguous and camera-angle sensitive.
  Overhang climbing violates the "elbows/knees bow away from wall" prior.
- Hold tagging is on a single reference frame; holds cannot be repositioned
  across frames (fine if the camera is mostly static — typical for a tripod).
- `MockPoseProvider` does not actually look at the video pixels; it emits a
  synthesized realistic motion trace. Real inference is one dev-build away.
- No cloud sync, no auth, no social features. This is deliberate — see the
  roadmap below.

## Roadmap

- Finish `TFJSPoseProvider` (MoveNet Lightning) with a dev-build target.
- Real SMPL fit for the body mesh panel (keeping the current stylized mesh
  as a fallback).
- Multi-frame hold tagging + optional hold auto-detection (YOLO fine-tune).
- Optional cloud sync behind a feature flag — the storage layer is already
  factored to accept a remote adapter without touching screens.
- Tutor mode: the coach walks you through a flagged mistake frame-by-frame.
